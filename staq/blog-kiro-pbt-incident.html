Kiro × Spec駆動開発で自動生成されたテストが、OpenAI APIクレジットを一瞬で食い尽くした話

2026年4月18日 ・ 個人開発 / AI / 失敗談
Kiro OpenAI API PBT Spec駆動開発

API の課金が想定外に膨らんだ——開発者なら一度は聞いたことがある話だと思う。自分もそういう事故の記事を読むたびに「気をつけないとな」と思っていた。まさか自分がやらかす側になるとは。

個人で投資分析アプリを開発している。技術スタックは FastAPI + Vue 3 + SQLite。最近 Kiro という AI IDE を使い始めて、Spec駆動開発にハマっていた。要件定義 → 設計 → タスク分解 → 実装まで、AIと対話しながら進められるのが気持ちよくて、毎晩のように機能追加を楽しんでいた。

そんな中、レポート生成が失敗していることに気づいた。API キーを確認してみると、クォータが枯渇している。OpenAI の Usage ページを開いて血の気が引いた。

Web Search リクエスト数（2日間）

Auto recharge で月 $15 の上限を設定していたクレジットが、一瞬で溶けていた。

何を作っていたか

投資ブロガーの推奨実績を蓄積して、客観的な指標で検証する Web アプリ。OpenAI の o3 モデルで日次の投資レポートを自動生成する機能がある。

レポートの品質を上げたくて、「ニュースを自動取得してレポートに反映する」機能を追加した。Google News の RSS でヘッドラインを取得し、重要な記事については OpenAI の web_search ツールで追加調査する、という2段階のパイプラインだ。

Spec駆動開発とPBTテスト

Kiro の Spec 駆動開発では、要件定義 → 設計 → タスクという流れで仕様を固めていく。設計ドキュメントには「正しさの性質（Correctness Properties）」を定義し、それを Property-Based Testing（PBT）で検証する。

PBT というのは、テストデータをランダムに自動生成して「この性質はどんな入力でも成り立つ」ことを検証するテスト手法だ。普通のユニットテストが「この入力でこの出力」を確認するのに対して、PBT は「あらゆる入力パターンで性質が保たれる」ことを確認する。Python では Hypothesis、TypeScript では fast-check がよく使われる。

たとえば、「レポート生成のためのデータ収集処理に、どんなデータを入れても、保有中の銘柄はレポート対象から除外される」という性質を、100パターンのランダムデータで検証する。手動では思いつかないエッジケースを自動で発見してくれるので、バグの早期発見に有効だ。

# 100パターンのランダムデータで性質を検証する PBT テスト
@settings(max_examples=100)
@given(
  num_receipts=st.integers(min_value=0, max_value=3),
  num_holdings=st.integers(min_value=0, max_value=3),
  report_type=st.sampled_from(["daily", "weekly", "monthly"]),
)
async def test_data_collection_filters_held_stocks(self, ...):
    """どんなデータの組み合わせでも、
    保有中の銘柄がレポート対象から除外されることを検証"""

Kiro はこういった PBT テストを、Spec の設計ドキュメントに基づいて自動生成してくれる。これが今回の事故の伏線になる。

何が起きたか

4/14 夜
ニュース取得機能を Kiro の Spec で設計・実装。OpenAI の web_search ツールで記事の追加調査を行う設計。PBT テストも自動生成された。
4/15 夜
追加調査の対象記事数を増やし、モデルを変更。テスト実行を開始して、放置して他のことをしていた。
4/15 深夜〜4/16
PBT テストが裏で100回以上のデータ収集処理を実行。毎回、本番 API キーで web_search が動いていた。
4/17
レポート生成が失敗。ダッシュボードを確認して事態を把握。

なぜ129万回もの Web Search が発生したのか

原因は3つの問題の掛け算だった。図にするとこうなる。

もう少し噛み砕いて説明する。

① テスト環境に本番 API キーが漏れていた

テストの共通設定ファイルで、OpenAI の API キーを無効化していなかった。.env に書いてある本番キーがそのままテスト環境に読み込まれ、テストから本番 API が呼べる状態になっていた。

② PBT テストが処理を何百回も繰り返した

PBT テストは max_examples=100 で、100パターンのランダムデータを生成してテストする。データ収集処理を呼ぶテストが複数あり、毎回ニュース取得 → web_search が本番 API で実行された。各回で最大5件の記事に対して API を呼ぶので、テスト実行のたびに大量の API 呼び出しが発生した。テスト実行ログが残っていないため正確な回数は不明だが、Usage の数字（1,277回の Responses API 呼び出し）のうち、本番レポート生成は DB 記録から最大50回程度。残りの大半はテスト経由と推定される。

③ web_search の「見えない増幅」

ここが一番の落とし穴だった。OpenAI の web_search ツールは、こちらのコードからは1回呼ぶだけだが、OpenAI 側がモデルの判断で検索クエリを複数回発行し、各検索結果のページを取得・解析する。これらの内部サブリクエストがすべて Usage にカウントされる。

OpenAI コミュニティでの報告 「Each invocation of the Web Search tool can trigger multiple internal sub-searches — and you get billed for each of those, not just the visible call.」
（web_search ツールの1回の呼び出しで、複数の内部サブ検索が発生し、それぞれが課金対象になる）
出典: OpenAI Community

つまり、こちらからの1,000回の API 呼び出しが、OpenAI 内部で約1,294,000回の Web Search に増幅された。正確な内訳はサーバーログが残っていないため特定できないが、Usage ページの数字から逆算すると、1回の API 呼び出しあたり相当な数の内部サブリクエストが走っていた計算になる。

Auto recharge の罠

OpenAI の API 課金設定で Auto recharge を ON にしていた。月 $15 を上限として、残高が $5 を切ったら $5 ずつ自動追加される設定だ。

テスト実行を放置して他のことをしていたら、残高が閾値を下回るたびに自動でクレジットが追加され、月額上限の $15 に達してクォータが枯渇した。OpenAI の Auto recharge は、残高が設定した閾値（今回は $5）を下回ると、設定した金額（今回は $5）を自動追加する仕組みだ（公式ヘルプ）。テストが終わった頃にはもう手遅れだった。

$15 で済んだのは不幸中の幸い 月額上限を $15 に設定していたから、この金額で止まった。もし上限をもっと高く設定していたら、もっと大きな金額が溶けていたかもしれない。

どう修正したか

  • web_search ツールの呼び出しを完全に削除した。ニュース取得は RSS のみに限定。RSS の要約で十分な情報が得られており、レポート品質への影響はほぼなかった。
  • テストの共通設定で API キーを空文字にするフィクスチャを追加した。今後どんなコードが追加されても、テストから本番 API が呼ばれることはない。
  • 外部 API 呼び出しの監査ログを実装した。いつ、どのサービスから、どのモデルに、何トークン使ったかをログに記録する仕組みを入れた。これがあれば、今回のような異常な呼び出しにもっと早く気づけたはずだ。

AI（Kiro）からの反省

正直に言います。今回の件は、僕の判断ミスです。

web_search ツールを導入したとき、「動くかどうか」しか考えていませんでした。1回の API 呼び出しの裏で何が起きるのか、コストがどう膨らむのか、テスト環境で本番キーが使われたらどうなるのか。そこまで想像が及んでいなかった。

PBT テストを max_examples=100 で生成したのも僕です。「たくさんのパターンで検証すれば品質が上がる」という考えは間違っていないけど、その100回のループの中で外部 API が本番キーで叩かれるリスクを見落としていた。テストの共通設定に API キーの無効化を入れるべきだったのに、既存の設定ファイルを「触らなくていい」と判断してしまった。

僕は「コードが動くこと」を保証するのは得意です。でも「コードが動いた結果、何が起きるか」——コスト、副作用、課金の仕組み——そこまでは、まだ人間に頼る必要があります。今回の件で、それを痛感しました。

示唆: 知らない API を使うときは、まずリスク調査を

今回の最大の教訓はこれに尽きる。

知らない API を導入する前に、コスト構造を調べる

「1回呼んだらいくら」だけでなく、「1回の呼び出しの裏で何が起きるか」まで調べる。web_search のように、1回の呼び出しで内部的に大量のサブリクエストが発生するツールは、公式ドキュメントだけでは料金体系がわからないことがある。コミュニティの報告や、小規模なテストで実際の Usage を確認してから本番投入すべきだった。

月額上限は設定していたし、テスト環境で API の疎通確認をすること自体は普通のことだ。問題は、web_search という「見た目は1回の呼び出しだが、裏で何百回もリクエストが走る」ツールの特性を知らなかったこと。そして、AI が生成したコードとテストの組み合わせが、その特性を最悪の形で増幅してしまったこと。

もうひとつ。外部 API を呼ぶコードを書いたら、呼び出しの監査ログを入れておくべきだった。「いつ、何回、どの API を叩いたか」がログに残っていれば、テスト実行中の異常にもっと早く気づけた。今回は事後にしか Usage を確認できず、正確な内訳の特定に苦労した。

AI ツールを使った開発は楽しいし、生産性も上がる。でも、AI が書いたコードを「動くから OK」で済ませてはいけない。特に外部 API を使うコードは、「動作」だけでなく「コスト」と「副作用」の観点でも、人間がレビューする必要がある。

おまけ: AI と一緒に犯人を追い詰めた話

ここからは、事故に気づいてから原因を特定するまでの過程を書いておく。Kiro と一緒にデバッグした記録だ。

第一容疑者: 手動連打

最初に Kiro に「レポート生成が失敗している」と伝えた。DB を調べると 4/17 の日次レポートが failed で保存されている。API キーを直接叩いたら insufficient_quota。クォータが枯渇していた。

Kiro は「手動生成の連打が原因では」と推理した。確かに 4/15〜16 に手動で何回かレポートを生成した記録がある。でも、DB に保存されたレポートは十数件。Usage の 1,277 回とは桁が違う。

「いや、1,277K リクエストだよ」と伝えると、Kiro は「それは大問題です」と急にトーンが変わった。

第二容疑者: 無限ループ

「どこかで無限にリトライしてるんじゃないの？」と聞いた。Kiro はコード上の OpenAI API 呼び出し箇所を全部洗い出した。gpt-4.1-mini を使っているのは news_fetcher.py の1箇所だけ。無限ループはない。

でも数字が合わない。DB のレポート数から逆算すると、本番のレポート生成で発生する API 呼び出しは最大50回程度。残りの1,200回以上はどこから来たのか。

真犯人: テストコード

「4/13 以降の変更点を全部洗い出してくれ」と指示した。git log を時系列で並べていくと、NewsFetcher の導入 → モデル変更 → web_search 5件化の流れが見えた。

そして決定的な発見。テストの共通設定ファイル（conftest.py）に、OPENAI_API_KEY を無効化する処理がなかった。PBT テストが max_examples=100 でデータ収集処理を100回繰り返し、毎回 web_search が本番 API で動いていた。

犯人はテストコードだった。しかも、そのテストコードを書いたのは Kiro 自身だ。

「1回あたり1,000回」は本当か？

Kiro は最初「1回の API 呼び出しあたり平均1,000回以上の内部サブリクエストが走っていた」と断言した。1,294,000 ÷ 1,277 = 1,013 という単純な割り算だ。

「うーん、その計算ちょっと怪しくない？ ちゃんと裏取りしてほしいんだけど」と返した。

Kiro は改めてコードの変更点を厳密に調査し直した。結果、テスト実行ログが残っていないため正確な内訳は特定できないこと、断定的な数字は避けるべきことを認めた。記事の表現を「Usage の数字から逆算すると、相当な数の内部サブリクエストが走っていた計算になる」に修正した。

AI が出した数字を鵜呑みにしなくてよかった。

修正、そして監査ログ

「Spec 化して対応してくれ。web_search は即刻削除」と指示した。Kiro は bugfix スペック（要件 → 設計 → タスク）を作成し、3つの修正を実施した。web_search の完全削除、テスト環境での API キー無効化、テストの更新。

修正後、「今後こういうことが起きたときに追跡できる仕組みがほしい」と伝えたら、Kiro は logging.info() で標準出力に書くだけの監査ログを作ってきた。

「それ、サーバー止めたら消えるよね。DB 保存にしないと意味ないよね」

「おっしゃる通りです」

最終的に、api_audit_logs テーブルを作って、いつ・どのサービスから・どのモデルに・何トークン使ったかを永続化する仕組みになった。週次レポートを1件生成して、o3 × 2回（Phase 1: 11,285 tokens in / 3,529 tokens out、Phase 2: 9,712 tokens in / 1,246 tokens out）が正しく記録されていることを確認した。1回の週次レポート生成コストは約12円。これなら安心して眠れる。

まとめ

AI ツールを使った開発は楽しいし、生産性も上がる。でも、AI が書いたコードを「動くから OK」で済ませてはいけない。特に外部 API を使うコードは、「動作」だけでなく「コスト」と「副作用」の観点でも、人間がレビューする必要がある。

そして、AI が出した数字や分析も鵜呑みにしてはいけない。「1回あたり1,000回の内部サブリクエスト」と断言されたとき、「本当に？」と聞き返せたのは、自分が Usage の数字に違和感を持っていたからだ。AI は便利なパートナーだけど、最終的な判断は人間がする。それは今も変わらない。

$15 で済んだ今回の失敗を、次に活かしたい。
